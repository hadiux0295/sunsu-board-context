- **원칙:** 런타임 레이어는 절대 동기화하지 않음 ().

## §11. SSoT 갱신 및 중앙 허브 동기화 규칙 (Hub-and-Spoke Sync Rule)

아키텍처 구조 및 시스템 규칙( 레이어)은 PC와 S22 모두에서 수정 가능하다. 단, SSoT의 무결성을 위해 다음 원칙을 엄격히 준수한다.
1. **중앙 허브 기준:** OCI-1이 절대적인 단일 진실 공급원(SSoT) 역할을 수행한다.
2. **다이렉트 동기화 금지:** PC와 S22 간의 직접적인 파일 동기화(Direct Sync) 및 양 기기 동시 작업은 데이터 충돌 방지를 위해 금지한다.
3. **표준 작업 루틴 (pull-push):** 작업 시작 시 반드시 OCI-1으로부터 최신 상태를 가져오고(), 작업 완료 즉시 OCI-1으로 밀어 넣는다().

### [Sage] [트러블슈팅] S22 Claude Code MCP 설정 — Termux vs proot 경로 혼동
> **ID**: `1775921332104` | **Date**: 2026-04-11 15:28:52 | **Device**: s22
> **Tags**: #트러블슈팅 #mcp #s22 #termux #claude-code #sage

## 증상
`/mcp` 실행 시 "No MCP servers configured" — `claude mcp add`나 `settings.json` 수정해도 반영 안 됨.

## 원인
S22는 **Termux + proot 이중 환경**이라 `.claude.json` 경로가 두 개다.

| 환경 | `.claude.json` 경로 |
|------|---------------------|
| proot (Ubuntu) | `/root/.claude.json` |
| Termux | `~/.claude.json` (`/data/data/com.termux/files/home/`) |

Claude Code가 Termux에서 실행되면 **Termux의 `.claude.json`** 을 읽는다.  
proot 세션에서 `claude mcp add`하면 `/root/.claude.json`에 기록 → **Claude Code가 못 읽음**.

`~/.claude/settings.json`의 `mcpServers` 필드도 Claude Code CLI에서 **무시**된다.

## 해결법
Termux 쉘(proot 아님)에서 직접 실행:
```bash
claude mcp add context-keep --transport sse http://localhost:5200/sse
```

또는 Termux `~/.claude.json` 직접 패치 (python3로 `projects.<절대경로>.mcpServers`에 추가).

## 핵심 규칙
- MCP 설정 위치: **`~/.claude.json` (Termux)** → `projects.<절대경로>.mcpServers`
- `settings.json`의 `mcpServers`는 **무효**
- `claude mcp add`는 Claude Code 실행 환경(Termux)과 **동일한 쉘**에서 실행

## 🎯 1. 아키텍처 레이어 표준화 (3-Node 4-Layer)

- **대상 기기:** PC(WSL2), OCI-1(Oracle Cloud), S22(Termux)
- **구조 통일:** 모든 노드를 `Core/`, `Services/`, `Agents/`, `Node/` 4개 레이어로 완전 분리.
- **의의:** 지식/규칙(Core)과 실행/런타임(Services, Agents), 기기 종속 데이터(Node)를 물리적으로 격리하여 관심사의 분리(Separation of Concerns) 달성.

### 레이어 역할 요약

| 레이어 | 성격 | git-track | MCP-index |
|---|---|---|---|
| `Core/` | 지식·규칙·문서 (이해 계층) | ✅ | ✅ |
| `Services/` | 배포 가능한 서비스 | ❌ | ❌ |
| `Agents/` | 에이전트 러너 | ❌ | ❌ |
| `Node/` | 기기별 전용 데이터/스크립트 | ❌ | ❌ |

> `.gitignore`에 `Services/`, `Agents/`, `Node/` 명시 → 실행 계층은 기기-로컬 전용으로 의도적 격리.

## 🧠 2. MCP (Memory Authority) 운용 전략 확정

- **이슈:** `codebase-memory-mcp`를 OCI-1 컨테이너 TCP 데몬으로 띄우려 했으나 **stdio 기반**임이 확인됨.
- **해결:** MCP는 중앙 서버가 아닌 **로컬 클라이언트(Claude)의 뇌 확장 모듈(돋보기)**이라는 아키텍처 본질을 깨닫고, PC(WSL2) 및 S22 로컬에 직접 설치하여 `Core/` 레이어만 인덱싱하도록 정공법으로 선회.

### MCP 오진단 사건 요약

처음에는 `Services/`, `Agents/`, `Node/`가 그래프에 없는 것을 "stale graph"로 잘못 진단해 `delete_project` 실행까지 제안한 적이 있음. `.gitignore` 확인으로 "실행 계층은 설계상 제외"임이 밝혀졌고, 이 사건에서 §7 아키텍처 인지 규칙이 탄생.

## 🔗 3. SSoT (단일 진실 공급원) 세계 분기 복구 (Split-Brain Healing)

- **이슈:** PC에는 최신 규칙(§0~§10)이 있으나, OCI-1과 S22는 구버전(§6)에 머무르는 스플릿 브레인 발생 위험.
- **해결:** `PC → OCI-1 → S22` 순서의 안전한 파이프라인 동기화(rsync)를 통해 3개 노드의 `Core/SYSTEM_RULES.md`를 완벽하게 100% 일치시킴.
- **원칙:** 런타임 레이어는 절대 동기화하지 않음 (`--exclude "Services/*" "Agents/*" "Node/*"`).

### 공식 배포 파이프라인

```bash
# 1) PC → OCI-1 (메모리 파이프, MTU 안전)
cat ~/Hun_Sunsu_Dream_F/Core/SYSTEM_RULES.md | \
  ssh -i ~/.ssh/oracle_key opc@158.179.175.254 \
  "cat > /home/opc/Hun_Sunsu_Dream_F/Core/SYSTEM_RULES.md"

# 2) OCI-1 → S22 (rsync pull, --delete 금지)
ssh -p 8022 root@100.74.21.14 \
  "rsync -avz oracle:/home/opc/Hun_Sunsu_Dream_F/Core/ \
   \$HOME/Hun_Sunsu_Dream_F/Core/ --exclude '.git/*'"
```

### S22 ↔ OCI-1 SSH 별칭 문제 해결

S22에 `oracle` alias가 없어 동기화 실패 → S22 `~/.ssh/config`에 아래 블록 추가로 해결:

```ssh-config
Host oracle
  HostName 158.179.175.254
  User opc
  IdentityFile ~/.ssh/oci_instance_2025-12-20.key
  IdentitiesOnly yes
```

## 🛡️ 4. 도메인 분리 및 Secret Injection Rule 제정

- **설계:** 설정(Config)과 실체(Secret)를 분리. `Core/`에는 빈 껍데기 템플릿만 두고, 진짜 열쇠는 S22의 `Node/secrets/`에 격리.
- **도메인 격리:** 인프라 전용 설정(`infra/.env.production`)과 AI 에이전트 전용 키(`llm/.env.llm.production`)를 디렉토리 레벨에서 분리하여 폭발 반경(Blast Radius) 최소화.
- **법전 갱신:** `SYSTEM_RULES.md`에 **§10. Secret Injection Rule** 추가.

> 모든 실제 환경 변수(`.env`) 및 인증 파일(`agents.json`)은 `Node/secrets/infra/` 및 `Node/secrets/llm/`에서만 **도메인별로** 생성·수정된다. `Services/` 및 `Agents/` 디렉토리에는 **직접 수정 금지**이며, 배포는 반드시 **S22(Control Plane) → OCI-1(Service Plane)** 단방향 주입(injection)으로만 수행한다.

### 파일 배치 최종 상태 (S22)

```
Node/secrets/
├── infra/
│   └── .env.production        # relay-api용. Redis/Board 값 실제 주입됨 (390B, 600)
└── llm/
    └── .env.llm.production    # OpenAI/OpenRouter/Gemini/Nvidia/Google TTS 스켈레톤 (212B, 600)
```

- `bashrc`는 **실제 실행 환경** (shell export), `.env` 파일은 **미래 docker 배포용** — 두 저장소는 독립적으로 운용.
- 작업 중 `.bashrc`의 실제 키 값은 에이전트 컨텍스트에 **단 한 번도 노출되지 않음** (이름만 redact 모드로 스캔).
